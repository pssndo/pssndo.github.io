Event A · Anatomy

What the cliff conserved — and what it didn't

apple_proxy share before vs after the cliff — per client (the settle ratio varies wildly)

Conserved invariants

  • Total Apple-family traffic per client per day is essentially flat across the cliff (within ±5% day-over-day noise). AtWave: ~3.4M/day before AND after.
  • Click attribution patterns — clicker device mix (desktop / phone), CTR per impression, click → impression impressionId joins all unaffected. Apple users were still Apple users; only their classification label changed.
  • Pre-cliff ratio was identical across all 5 clients: 93–95% apple_proxy.

What shifted

  • The within-Apple split changed per-client by different amounts: AtWave settled at 40%, Newcom at 87%. The post-event ratio depends on which specific Apple IP ranges the client's recipients sit on.
  • Jun 9 was the worst day — almost all Apple traffic classified as apple_mail. Jun 10 onward partially reverted to a new equilibrium.
  • The "MPP enabled vs not" interpretation we'd been ascribing to the split is wrong. The labels are a function of IP-intel state, not user behavior — Apple's MPP enrollment doesn't change overnight by ~70pp.
Operational consequence: any report or dashboard that groups by deviceId over a window spanning Jun 9 produces misleading trend lines. The fix is upstream — re-aggregate to a synthetic Apple-family combined bucket (apple_proxy + apple_mail) before producing time series.
Event A · Where to look

The classifier logic and likely root cause

The classifier (current state)

// backend/apps/adserver/src/Device/CreateDeviceUsingParser.php

private function isAppleProxy(string $userAgent): bool
{
    return $userAgent === 'Mozilla/5.0';
}

private function isAppleMpp(?string $vpnDescription, ?string $vpnType): bool
{
    return $vpnDescription === 'apple'
        && $vpnType === 'consumer-privacy';
}

The bare-Mozilla/5.0 UA matches Apple traffic. The MPP-vs-not distinction depends entirely on whether the upstream VPN/IP-intel resolver returns vpnDescription = 'apple' AND vpnType = 'consumer-privacy' for the source IP.

Suspect cause

  • IP intel data refresh from the VPN classifier provider (MaxMind, IP2Location, IPQualityScore, etc.). On Jun 8–9 the provider may have shipped an update that reclassified a large chunk of Apple's MPP infrastructure IPs out of the consumer-privacy bucket.
  • Or a code change in CreateDeviceUsingParser.php or its VPN-data resolver. Less likely given the cross-client uniformity, but worth checking the deploy log for Jun 8 UTC.
  • Or both — a deploy that reloaded VPN-intel data in a way that exposed a pre-existing pending update.

Why the partial revert on Jun 10?

Jun 9 was the extreme (~22% apple_proxy). Jun 10 onward settles higher. Either a fix went in (partial), or the IP-intel data continued updating asynchronously after the initial deploy.

Event B · Cross-client & impact

Yahoo's drift is platform-wide — and breaks any cross-window calibration

Cross-client Yahoo clean share (Jun 1–15)

Yahoo clean % — same window, 5 clients

All 5 clients cluster at 55–67% clean in the same window — confirming the drift is platform-level, not client-specific.

Impact on the Yahoo-as-anchor methodology

The AtWave real-impressions analysis used Yahoo's clean subset as ground-truth ("clean ≈ 100% real opens") to derive a +23% calibration constant on the click-anchored real-share estimator. That number is correct for the May 22 – Jun 7 regime (Yahoo at 55%), but:

  • In the Mar 15 – Apr 28 regime (Yahoo at 97%), the calibration would be near zero — the click-anchored method was already roughly right.
  • In the Apr 29 – May 20 regime (Yahoo at 87%), the calibration would be smaller — perhaps +10%.
  • The +23% from June can't be carried backward without producing wrong answers.
Methodology rule: any calibration constant derived from a Yahoo-subset analysis must be re-computed inside the same window as the data being analyzed. Cross-window comparisons need their own per-window calibrations.
Operational impact

What these silent reclassifications break

Per-segment trend reports

Anything grouping by deviceId over a window spanning Jun 9 shows a "real" but spurious shift. Customer-facing dashboards using apple_proxy / apple_mail split are misleading without aggregation to Apple-family.

Fraud / billing logic

If any downstream uses fraudType 3 (HOSTING) as a discount factor (e.g. observe-mode exclusion), the rolling Yahoo drift means clients' billed counts shift silently with each IP-intel step. Need to check whether any of our existing aggregate MVs depend on this.

Calibration constants

Per the AtWave methodology: any "drift" measured from a Yahoo back-check is window-specific and must be re-derived. Old constants applied to new data produce wrong real-share estimates.

Cross-window analyses

Backtests, A/B test readouts, and any longitudinal report that grew its window across one of these dates needs a regime-aware aggregation function — not a simple GROUP BY deviceId.

Customer questions

"Why did my apple_proxy traffic drop on Jun 9?" is now a recurring CSM question we have no current automated answer for. Knowledge of the cliff lives in this deck and three markdown reports.

Future events

If IP-intel data updates monthly (Yahoo's pattern suggests it does), expect ~12 of these per year. Each silent. Each can shift report shapes by 10–30pp.

Recommendations

What to build / change

Detection (low effort, high value)

  • Daily classification-ratio Sentry breadcrumb. Compute the per-publisher daily share of each deviceId and each fraudTypeId. Emit a Sentry breadcrumb when any segment's share shifts more than 20pp day-over-day. One-shot scheduled job against the aggregate MVs; ~10 lines of code.
  • Weekly IP-intel data version log. Whatever VPN/IP intelligence provider feeds vpnDescription / vpnType: capture and persist the provider's data version / publish date on every refresh. Lets us correlate future shifts with intel updates.
  • Deploy log enrichment. Any deploy that touches CreateDeviceUsingParser.php or the VPN-data loader gets a one-line note + a marker in our metrics dashboard. Free for future investigators.

Reporting hygiene

  • Treat apple_proxy + apple_mail as one Apple-family unit in all longitudinal reports and external dashboards. The sub-label is IP-intel state, not user behavior.
  • Document the Apr 29 / May 21 / Jun 9 regime boundaries in the analytics docs (docs/overview/) so future analyses don't accidentally span them with a single GROUP BY.
  • Calibration constants must be window-bound. Any derived multiplier (like the AtWave +23%) carries a "valid for window X→Y only" tag.

Investigation (one-shot)

  • Confirm the Jun 8–9 root cause. Pull the git log + deploy log for Jun 7–9 and any IP-intel data refresh records. If the cause was a vendor update, file a ticket; if it was our code, decide whether to revert / pin / accept.
  • Spot-check the Apr 29 and May 21 events on at least one other client to confirm they're platform-wide (high confidence, but not yet verified — only AtWave was queried at daily granularity for that period).
Reproducibility

SQL anchors — re-run any of this

Apple cliff per client (daily)

SELECT
  toDate(datetime) AS day,
  countIf(deviceId = '8c02db85-d54c-41a2-ab75-5db20a9f6f3e') AS apple_proxy,
  countIf(deviceId = 'ae34b6a0-726e-4e9a-825f-ace9c1708bab') AS apple_mail,
  round(countIf(deviceId = '8c02db85-d54c-41a2-ab75-5db20a9f6f3e') /
    nullIf(countIf(deviceId IN ('8c02db85-...','ae34b6a0-...')), 0) * 100, 1) AS proxy_share_pct
FROM emperor.events
WHERE publisherClientId = '<client_id>'
  AND datetime >= toDateTime('2026-05-25 00:00:00')
  AND datetime <  toDateTime('2026-06-15 00:00:00')
  AND eventType = 'IMPRESSION' AND countableStatus = 1
  AND deviceId IN ('8c02db85-...','ae34b6a0-...')
GROUP BY day ORDER BY day

Yahoo clean/hosting timeline per client (daily)

SELECT
  toDate(datetime) AS day,
  countIf(fraudTypeId = 60) AS clean_60,
  countIf(fraudTypeId = 3)  AS hosting_3,
  round(countIf(fraudTypeId = 60) / nullIf(count(), 0) * 100, 1) AS clean_pct
FROM emperor.events
WHERE publisherClientId = '<client_id>'
  AND datetime >= toDateTime('2026-04-20 00:00:00')
  AND datetime <  toDateTime('2026-06-12 00:00:00')
  AND eventType = 'IMPRESSION' AND countableStatus = 1
  AND deviceId = 'fd7db5aa-8c9d-4595-8116-152c9b1acd30'  -- yahoo proxy
GROUP BY day ORDER BY day

Note: emperor.events is sorted by datetime only; per-day GROUP BYs over 90 days run cleanly under the 30s server budget. Adding a high-cardinality dimension (e.g. userIdentity) tips into timeout — chunk by week or use the aggregate MVs where possible.
