Framework

The right shape of a sendout identifier

What it should be: the same ID for every email in one blast cohort. If a daily newsletter goes to 50,000 subscribers on 2026-04-22, all 50,000 should share one identifier. Uniqueness per blast, not per recipient.

Diagnostic: events per sendout ID

Given a 7-day ClickHouse sample, we measure users ÷ distinct sendout IDs. A healthy cohort ID shows hundreds of users per ID. A per-recipient token shows ≈ 1.

What sendouts/ID tells us

Uniqueness ≠ correctness

A daily newsletter that's been running 6 months has roughly 130 sendout rows in MySQL. If the identifier cohorts correctly, there should be many sendouts sharing one ID (same blast wave seen via multiple delivery pulses, multiple templates, etc. → same cohort). Instead:

Sendouts per distinct ID (MySQL) — high is healthy cohorting, 1.0 is per-sendout unique
1.0 (pure unique) — identifier is unique for every newsletter_sendouts row. Sometimes the blast really does have 1 row per cohort (Adopstar: 1 newsletter, 1 row per day), but when combined with low CH-users/ID, it's per-recipient token leakage.
> 2.0 (cohorting) — multiple sendout rows share an ID because the blast was logged multiple times (e.g. test + live, or same date across newsletters). Newcom (8.9), Sosland (4.2), The Drum (1.8).

Mitigation applied ✨

Real blast cohorts after compute_cohort_id()

Applying the per-client trust policy from SENDOUT_IDENTIFIER_MITIGATION.md: trust the identifier for healthy clients, fall back to (newsletter_id, time-bucket) for broken clients or broken-literal rows.

Why raw ≈ est. cohorts on the MySQL side, and why that's expected:
newsletter_sendouts is already mostly one row per blast — the ingest pipeline deduplicates. So the MySQL row count is close to reality already. The real wins of the mitigation are:
  • Correctly counting distinct cohorts — today's COUNT(DISTINCT identifier) gives 46 for Newcom (looks like 46 blasts) when the real answer is 397. A bad metric masquerading as a good one.
  • Collapsing broken-literal rows — Newcom's 29 sendouts with identifier 2222 become 29 distinct cohorts again (via time-bucket fallback), not 1.
  • Massive CH-side reduction — where per-recipient tokens balloon cardinality 95-400×, and where our aggregate MVs actually live.

Client-level notes

What each client's data tells us

Key Media · Hubspot

5,299 sendouts · 110 newsletters · 23.9 users/ID

Identifier is an ISO-8601 datetime URL-encoded (2026-04-22%2011:50:07). Looks like a per-blast timestamp, and CH confirms thousands of users share each. But MySQL sendouts-per-ID = 1.0 (every row unique) means the timestamp goes down to the second — so two identical cohort pulses even 1s apart would get different IDs. Works as a cohort key in practice, but it's time-of-blast, not a stable "this is sendout #X of newsletter Y".

How this was computed

Methodology

Data sources

  • MySQL newsletter_sendouts — what Passendo stores. identifier is the parsed sendout key + datetime, created_at.
  • ClickHouse emperor.events — what the ad server receives. sendout column is the raw string from tracking URLs, unparsed.

Classification

Regex-based format classifier ran against all sendouts (7,430 rows) from the last 6 months:

CASE
  WHEN identifier = '2222' THEN 'recycled-placeholder-2222'
  WHEN identifier LIKE '1970-01-01%' THEN 'epoch-zero-bug'
  WHEN identifier LIKE '*|CAMPAIGN_UID|*' THEN 'unrendered-campaign-uid'
  WHEN identifier REGEXP '^[0-9]{4}-[0-9]{2}-[0-9]{2}%20' THEN 'urlencoded-datetime'
  WHEN identifier REGEXP '^[0-9]{4}-[0-9]{2}-[0-9]{2} '   THEN 'plain-iso-datetime'
  WHEN identifier REGEXP '^[a-z_]+_[0-9]{4}-[0-9]{2}-[0-9]{2}_' THEN 'braze-name-date-time'
  WHEN identifier REGEXP '^[0-9]{8}$' THEN 'numeric-8'       -- MMDDYYYY or DDMMYYYY (disambiguated manually)
  WHEN identifier REGEXP '^[0-9]{6}$' THEN 'numeric-6'
  WHEN identifier REGEXP '^[0-9]+\\.[0-9]+$' THEN 'numeric.numeric'
  WHEN identifier REGEXP '^[a-f0-9]{10}$' THEN 'hex10-token'
  WHEN identifier REGEXP '^[0-9]+$' THEN 'numeric-var'
  ELSE 'other'
END

Health metric — the key diagnostic

From ClickHouse events (7-day window): avg_users_per_id = uniqExact(userIdentity) / uniqExact(sendout). Classified:

  • good ≥ 100 users/ID — one ID per blast cohort
  • warn 10–100 users/ID — partial leakage or very fine time granularity
  • bad < 10 users/ID — per-recipient token or broken identifier

DDMMYYYY vs MMDDYYYY disambiguation

For numeric-8 formats, a regex can't distinguish DDMMYYYY from MMDDYYYY when day ≤ 12 (e.g. 01082025 = Aug 1 or Jan 8?). Disambiguated by checking sendouts where the first two chars > 12 — if ANY exist, it's DDMMYYYY. The Drum has many (UK format); Newcom/Sosland have none (US MMDDYYYY).

Bottom line

Summary of what we learned

  • Format diversity is real: 10 distinct identifier formats across 12 clients. Date-in-identifier (6 clients), opaque campaign ID (4), short hash (1), and free-text/broken tokens (many).
  • No client has ever switched formats. Changes are additive — new formats appear alongside old ones, typically because a new newsletter launches with a different ESP config. Historical data joins are safe per-client-per-newsletter.
  • 100% identifier uniqueness in MySQL is a red flag, not a feature. The healthy pattern is "one ID per blast cohort" → many sendout rows share one ID. Newcom (8.9 sendouts/ID) and Sosland (4.2) exemplify this. Clients at 1.0 sendouts/ID need a second check against CH users-per-ID.
  • 4 clients are currently emitting per-recipient tokens: Are Media, Adopstar, Realestate.co.nz, 1440 — all with < 1 CH user per ID. Their sendout identifiers cannot be used as cohort keys.
  • 15 distinct unrendered template tokens across 6 clients in the last 14 days alone. Most severe: Newcom 2222 (234k events) and Key Media {{local_dt}} (111k events).
  • Realestate.co.nz stopped sending 2026-02-04 and has been silent for 2.5 months. Needs investigation.
  • Adopstar is an integration testbed, not a production data source. Exclude or flag.
✨ With the mitigation layer applied:
  • Every client gets a reliable blast-cohort key (sendout_cohort_id) — 7,430 MySQL sendouts → 7,273 real blast cohorts across 12 clients.
  • Per-recipient token explosion is collapsed: 95× to 387× reduction in CH cohort cardinality for the 4 broken clients.
  • Every cohort carries a confidence tag (identifier, identifier-plus-time, time-only) so reporting can be honest about precision.
  • Historical data is recoverable — backfill takes seconds per client because the derivation is deterministic from existing columns.
