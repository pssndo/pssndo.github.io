📖 Overview

Introduce a Feature Bundle system allowing superadmins to define named groups of features and assign them to clients. A client's resolved feature set is the union of all features across their assigned bundles.

💡 Concepts

Term Description
Feature A database-managed object with UUID id, human-readable featureId (e.g. TARGETING.IP_ADDRESS), name, and optional description. Represents a capability gate in the app.
Bundle A superadmin-managed collection with UUID id, name, optional description, and array of full Feature objects.
Resolved Features The deduplicated union of all features from all bundles assigned to a client. Always returned as full Feature objects across all endpoints.

🏗️ Data Models

Feature

interface Feature { id: string; // UUID featureId: string; // "TARGETING.IP_ADDRESS" name: string; // "IP Address Targeting" description?: string; // Optional }

Bundle

interface Bundle { id: string; // UUID name: string; // "Enterprise Tier" description?: string; // Optional features: Feature[]; // Full objects }

Client (CRUD)

interface Client { // ... existing fields bundles: Bundle[]; // Full Bundle objects features: Feature[]; // Resolved features }

Client (/me)

interface ClientMeResponse { // ... existing fields features: Feature[]; // Full Feature objects }

🔌 Backend API

Features Endpoints

GET POST /api/v3/features

List all features / create a new feature.

  • Full CRUD support (no longer hardcoded)
  • GET Response: { features: Feature[] }
  • POST Body: { featureId: string, name: string, description?: string }
GET PUT DELETE /api/v3/features/:id

Retrieve, update, or delete a feature (UUID).

⚠️WARNING

Update and delete operations are dangerous as they affect app-wide feature gates. Frontend must show strong confirmation UI before allowing PUT/DELETE operations.

Bundles Endpoints

GET POST /api/v3/bundles

List all bundles / create a new bundle.

  • GET Response: { bundles: Bundle[] } (with full Feature objects)
  • POST Body: { name: string, description?: string, features: Feature[] }
GET PUT DELETE /api/v3/bundles/:id

Retrieve, update, or delete a bundle.

Client Endpoints

GET PUT /api/v3/clients/:id

Full Client Response with nested objects.

  • bundles: Array of full Bundle objects
  • features: Array of full Feature objects (resolved)
GET /api/v3/me

Current authenticated client.

  • features: Array of full Feature objects
// Feature gate check example const { features } = useAuth(); // from /api/v3/me if (features.some(f => f.featureId === 'TARGETING.IP_ADDRESS')) { // show IP targeting UI }

⚠️CRITICAL: Feature Edit/Delete Confirmation

Before allowing PUT or DELETE on features, show a confirmation modal with:

"I have verified with backend team that this change is safe"
"I have verified with frontend team that this change is safe"

🏷️ Naming Conventions

UUIDs vs Human-Readable IDs

Features

  • id — UUID (database)
  • featureId — Const format (e.g., TARGETING.IP_ADDRESS)
  • Used for feature gate checks in code
  • Cannot be changed after creation

Bundles

  • id — UUID (database)
  • name — Human-readable (e.g., "Enterprise Tier")

Feature ID Format

TARGETING.IP_ADDRESS / GEO / DEVICE / ... TIER.LITE / STANDARD / PLUS / SCALE / ADVANCED / ENTERPRISE ADDON.PRO_CREATIVES / AUDIENCE_INTEL / EXT_DEMAND / ... CLIENT.{SLUG} BETA.{FEATURE} EXP.{EXPERIMENT_NAME}

Namespaces

Namespace Purpose
TARGETING.* Targeting capabilities
TIER.* Pricing/access tiers
ADDON.* Optional add-on features
CLIENT.* Client-specific custom features
BETA.* Beta/experimental features
EXP.* A/B test experiments

Naming Examples

  • Feature names: "IP Address Targeting", "Enterprise Tier", "Pro Creatives"
  • Bundle names: "Enterprise Tier", "Plus Tier", "Pro Creatives Add-on", "ACME Corp Custom"
