The question

Do we have a usable window to read the email HTML before ads serve?

If the answer is yes, we can run topic extraction, brand-safety checks, sentiment scoring, or audience-matching on the email body and feed the result into ad selection — per-sendout, in real time.

If the answer is no, we need a different ingest mechanism or we accept post-hoc enrichment with caching.

Data model at play

MySQL: newsletter_sendouts
  • datetime — ESP-reported send time
  • created_at — row insert time (proxy for "Passendo knows this sendout")
  • body — the rendered email HTML (50–150 KB)
  • identifier — ESP-specific sendout key (maps to CH events.sendout)
ClickHouse: emperor.events
  • datetime — event time (UTC)
  • newsletterId, publisherClientId — MySQL UUIDs verbatim
  • sendout — matches newsletter_sendouts.identifier for most ESPs
  • 2.3 B rows, 132 GB compressed — query carefully

Metric of interest

created_lead = first_event_ts − created_at
send_lead    = first_event_ts − datetime

Positive created_lead = content in MySQL before serving started ✅
Negative created_lead = we served blind ❌

Sanity check

datetime is reliable; created_at is not

The same delta measured against newsletter_sendouts.datetime (ESP-reported send time) instead of created_at sits within ±60 s of the first CH event for every client. That tells us:

  • datetime is a reliable "send happened at" signal — it matches reality within a minute.
  • created_at is pure insert-side lag: the row lands in MySQL 2–6 minutes after the send actually happened.
  • So the question isn't "when did we know about the send?" — it's "when was the body populated?". And the answer is: minutes too late.

ESP behavior

Lag is ESP-specific but always positive

Omeda (Newcom, Sosland)

Consistent 4–5 min lag, very low variance. Looks like a scheduled post-send hook firing on a timer rather than an event-driven insert.

Hubspot (Key Media)

Widest variance (−361 to −32 s). Probably event-driven but queue-depth-sensitive — lag scales with how busy Hubspot's webhook fleet is.

Sailthru (Are Media)

Tight −288 to −297 s body, one outlier at −6 s. Fast post-send hook but still post-send. The −6 s case barely beat the first impression.

Multi-ESP (Adopstar, Gloo)

Varies by routing. Gloo's −87 s median is the best-performing in the entire dataset, but still negative.

Salesforce (Conexus)

Small N (3) but exceptionally consistent: −260 to −264 s. Probably a fixed post-send delay.

General/SF2 (Momentum, Drum)

Mid-range lag (−87 to −356 s). Bespoke integrations; behavior depends on the specific send path.

The pattern is universal. Across every ESP, every timezone, every newsletter frequency — the row is inserted after the send. The specific latency differs, but the direction doesn't.
Implication

Content-aware ad selection: three paths forward

Option 3 — Cheapest

Cache yesterday's enrichment

Recurring newsletters (CU-Daily, Adv-AM etc.) have near-identical templates day-to-day. Enrich yesterday's sendout, store topic/brand-safety flags, reuse for today's first impressions. Wrong only on layout-change days (rare).

~95% of the value, ~5% of the work

Option 1 — Right long-term

Capture HTML at ingest, not post-send

Wherever the adapter receives the ESP's template request, persist the HTML there — into a new pre-send content store (Redis/Postgres) keyed by (newsletter_id, sendout_identifier). Keep newsletter_sendouts insert as-is for reporting.

Requires backend change + ESP-adapter audit

Option 2 — ESP-dependent

Pull rendered HTML from ESP pre-send

Some ESPs expose rendered HTML before physical send: Omeda's template preview API, Sailthru's template endpoint. Wire this in per-ESP. High effort, per-ESP integration.

Best coverage but expensive to build

Caveats

Data-quality flags worth fixing

1. Per-user sendout tokens (needs template review)

Some clients emit per-recipient tracking tokens in events.sendout, inflating cardinality in CH:

Distinct sendout values in CH vs actual sendouts in MySQL

Are Media: 703k CH values ÷ 43 MySQL sendouts = 16,349× per-user inflation. Each token binds to one user (uniqExact(userIdentity)=1). Adopstar, Key Media (partial) affected too. Fix: normalize the tracking tag to match newsletter_sendouts.identifier.

2. ESP-specific identifier format variance

ESPExampleMatches CH sendout?
Omeda04212026✅ exact
Sailthru45335918.1⚠️ per-user suffix divergence
Hubspot2026-04-22%2011:50:07⚠️ URL-encoded timestamp; some user-level spread
Salesforce153778⚠️ short numeric, collision risk across newsletters

Never string-match on sendout across clients. Always join on (newsletterId, time-window).

3. Known template bugs

  • Newcom (CU-Marsh) has events where sendout = '@{mv_date_MMddyyyy}@' — literal Omeda template token, never rendered. 291 impressions + 5 clicks leaked to this bad identifier. Filter with NOT startsWith(sendout, '@') or fix upstream.
  • All timestamps confirmed UTC on both sides (@@session.time_zone = SYSTEM, server in UTC). Delta is TZ-safe.
Methodology

How this was computed

Step 1 — Active clients

SELECT c.id, c.name, c.esp, c.timezone,
       COUNT(DISTINCT ns.id) AS sendouts_last_7d
FROM clients c
JOIN brands b ON b.client_id = c.id
JOIN newsletters n ON n.brand_id = b.id
JOIN newsletter_sendouts ns ON ns.newsletter_id = n.id
WHERE c.status = 'LIVE'
  AND ns.datetime >= (NOW() - INTERVAL 7 DAY)
GROUP BY c.id HAVING sendouts_last_7d > 0;

Step 2 — Per-sendout MySQL summary

SELECT b.client_id, ns.newsletter_id, DATE(ns.datetime) AS d,
       UNIX_TIMESTAMP(MIN(ns.created_at)) AS cts,
       UNIX_TIMESTAMP(MIN(ns.datetime))   AS sts,
       AVG(LENGTH(ns.body))               AS avg_bytes
FROM newsletter_sendouts ns
JOIN newsletters n ON n.id = ns.newsletter_id
JOIN brands b      ON b.id = n.brand_id
WHERE ns.datetime >= NOW() - INTERVAL 7 DAY
GROUP BY b.client_id, ns.newsletter_id, DATE(ns.datetime);

Step 3 — First CH event per (newsletter, send_ts) via arrayJoin boundary injection

WITH boundaries AS (
  SELECT tupleElement(t,1) AS nl, tupleElement(t,2) AS send_dt
  FROM (SELECT arrayJoin([ (nl, toDateTime(ts)), ... ]) AS t)
)
SELECT b.nl, toUnixTimestamp(b.send_dt) AS send_ts,
       toUnixTimestamp(min(e.datetime)) AS first_event_ts
FROM boundaries b
LEFT JOIN (
  SELECT newsletterId AS nl, datetime
  FROM emperor.events
  WHERE publisherClientId IN (<9 UUIDs>)
    AND datetime >= now() - INTERVAL 8 DAY
) e ON b.nl = e.nl
WHERE e.datetime >= b.send_dt - 60          -- tolerate early bot prefetch
  AND e.datetime <  b.send_dt + 21600       -- within 6h after send
GROUP BY b.nl, b.send_dt;

Step 4 — Python computes deltas, per-client aggregates, global histogram

created_lead = first_event_ts − created_ts
send_lead    = first_event_ts − send_ts
First attempt failed: originally joined on (newsletterId, toDate(datetime)). For non-UTC clients, UTC-day boundaries don't line up with local sendout days → produced nonsense ~12h deltas. Fixed by switching to a time-window join on (send_ts − 60 s, send_ts + 6 h).
Beyond this question

Playbook lessons for cross-DB analysis

Captured in docs/overview/CROSS_DB_ANALYSIS_PLAYBOOK.md for future sessions. The highlights:

The good

  • arrayJoin([...]) for boundary injection is the right pattern for "for each MySQL row, find the matching CH event" when MySQL side ≤ 1000 rows.
  • MySQL and CH timestamps are both UTC on this stack → delta is directly unit-safe.
  • UUIDs propagate verbatim MySQL → CH → events. Join keys align.
  • Use quantile(0.5)() and count()-style aggregates server-side so responses stay < 10 rows per client.

The bad & the ugly

  • Response-size overflow — hit repeatedly when pulling per-row MySQL + JSON format. Fix: markdown-table output and aggregate-only columns.
  • sendout identifier chaos — 5 ESPs, 4 formats, 2 of them per-user-explosive. Never string-match on it across clients.
  • UTC-date join trap — clients in non-UTC timezones cross the UTC midnight, making toDate()-based joins wrong by up to 12 hours.
  • The empty MySQL activity_log — Spatie convention places it in MySQL; real activity log is in clickhouse.emperor.activity_log (97k rows).
  • 1×1 fallback line item — ce29048b-... fires on unsold inventory with fraudTypeId ≠ 0. Filter countableStatus = 1 when counting "real" serving.
  • dictGet('emperor.publisher', ...) — readonly role lacks grant. Use federated emperor.clients instead.
Bottom line

Ship Option 3 now. Build Option 1 for Q3.

  • The premise doesn't hold today. Across 386 sendouts, 9 clients, and 5 ESPs, the newsletter_sendouts row lands after the first ad impression — 100% of the time, with a 4-minute median lag.
  • datetime is trustworthy; created_at is not. Use the former to anchor "when the send happened". Don't use the latter as a content-ingest signal — it's a post-send artifact.
  • The cheapest viable path is caching. Recurring newsletters have stable templates day-to-day. Pre-enrich yesterday's sendout, reuse for today.
  • The right long-term fix is a dedicated pre-send content store populated by the ESP adapter, separate from the reporting-focused newsletter_sendouts insert.
  • Flag for engineering: Are Media and Adopstar's per-user sendout-token behavior is inflating CH cardinality massively and should be normalized at the tracking-template level.
