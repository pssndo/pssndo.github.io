What Is This?

Emperor currently serves ads in newsletters, but it has no understanding of what the newsletter is about or how well an ad matches the content. A sports ad is just as likely to appear in a cooking newsletter as in a sports one (if targeting allows both).

This plan adds an AI layer that reads every newsletter, understands its content, analyzes every ad creative (text and images), and scores how well each ad matches each newsletter. The result: smarter ad placement, better click rates, and actionable insights for publishers.

Key discovery: The data we need already exists. Every ad impression in Emperor already records which newsletter sendout it came from. We don't need to change the data collection pipeline at all -- we just need to build new analysis on top of what's already there.

Who benefits?

See how each newsletter send performs, which ads match their content best, and get AI-powered recommendations to improve engagement -- all through dashboard widgets and reports.

💼

Delivery methods

  • Scheduled reports -- weekly email digests with AI-generated narratives summarizing performance
  • On-demand analysis -- publishers can request deeper analysis of specific sendouts or campaigns

How It All Fits Together

The system lives entirely within the existing Emperor backend (Laravel). There is no separate AI server to manage -- we call the Claude API directly over the internet, just like calling any other web service.

The three data stores

MySQL

Stores AI analysis results, relevance scores, insights, and creative content profiles. This is "what the AI thinks" about each newsletter and ad.

ClickHouse

Stores sendout performance statistics (impressions, clicks, revenue) at hourly and daily granularity. This is "how things actually performed." Optimized for fast aggregation queries.

Aerospike

In-memory cache for real-time data. Stores pre-computed match scores so the ad server can look them up instantly at serve time without calling the AI.

The AI: Claude API

Instead of running our own AI server (which would cost $1,300-2,500/month in GPU rentals plus maintenance), we use Anthropic's Claude API. It's a pay-per-use service -- we send text and images, get structured analysis back.

Why not self-host? Self-hosting an AI model like Qwen3 72B requires renting expensive GPU servers ($1,300-2,500/month), managing model deployments, handling scaling, and monitoring uptime. The Claude API eliminates all of that -- we just pay for what we use, starting at ~$180-250/month. That's 80-91% cheaper with zero infrastructure overhead.

Two models for different jobs

ModelUsed ForWhy
Haiku Content analysis, creative indexing, relevance scoring, anomaly detection, content-creative matching Fast, cheap, great at structured data extraction. Handles 95%+ of the workload.
Sonnet Insight generation, weekly report narratives, complex recommendations More capable at nuanced writing and multi-data-point synthesis. Used sparingly for quality.

Smart cost-saving techniques

Structured Output

We tell Claude exactly what format to respond in (JSON with specific fields). This eliminates parsing errors and wasted tokens on free-form text.

Prompt Caching

Instructions that don't change between calls (like "here are the IAB categories" or "score on a 0-1 scale") are cached by Anthropic. Cached tokens cost 90% less. We estimate ~40% of our input tokens will be cache hits.

Batch API

For work that doesn't need instant results (overnight analysis, weekly reports), we use the Batch API which processes requests within 24 hours at 50% the normal price.

Graceful Degradation

If the AI is down or rate-limited, nothing breaks. Ad serving continues normally. Analysis jobs queue up and retry automatically with increasing delays (3 attempts, then re-queued). The AI is an enhancement, not a dependency.

Budget protection

Monthly API spend is tracked. If spend exceeds a configurable threshold, non-critical AI jobs (insights, reports) are automatically paused while core analysis continues. Error rates and response times are monitored, with alerts triggered if sustained error rates exceed 5%.

How Newsletter Analysis Works

Every time a newsletter is sent (a "sendout"), the system automatically analyzes its content to understand what it's about.

What the AI extracts from each newsletter

🏷

Topics

What the newsletter is about, with confidence scores. e.g., "professional sports" (94%), "fitness equipment" (72%), "athletic apparel" (68%)

👥

Entities

Named things mentioned -- brands, people, organizations. e.g., "NBA" (organization), "LeBron James" (person), "Nike" (brand)

😊

Sentiment

The overall tone -- positive, negative, neutral, or mixed. A score from 0 to 1 (0.78 = positive).

📑

Summary + Keywords

A brief AI-generated summary and extracted keywords. e.g., "NBA playoffs", "basketball", "sports gear", "championship"

📂

IAB Category

Industry-standard content classification. e.g., "IAB17-12" = Sports > Basketball. Used for matching against ad categories.

📈

Readability Score

How complex the writing is. Helps correlate content complexity with engagement patterns.

Example output: A sports newsletter about NBA playoffs would be tagged with topics like "professional sports" and "basketball," entities like "NBA" and "LeBron James," positive sentiment (score 0.78), summary covering playoff highlights, and IAB category "Sports > Basketball." This profile is then stored and used to match ads.

How Ad Creative Indexing Works

Before we can match ads to newsletters, we need to understand what each ad is about. All ~10,000 active creatives are analyzed and indexed using Claude's ability to understand both text and images.

This runs in the background, not during ad serving. Creative indexing is a batch process. It happens when a creative is created or updated, and as a nightly catchup job. The ad server never waits for AI -- it uses pre-computed results.

What the AI extracts from each ad creative

Extracted DataDescriptionExample
TopicsWhat the ad is about"sports gear", "athletic apparel"
Product categoriesWhat's being sold"footwear", "activewear"
IAB categoriesIndustry-standard ad classification"IAB17-12", "IAB17-18"
EntitiesBrands, products, people"Nike", "Air Max"
ToneThe ad's mood/approach"promotional", "informational", "urgent"
Content summaryAI description of the ad"Nike running shoes promotion with athlete imagery"
Visual elementsWhat's in the image"product shot", "lifestyle photo", "text overlay", "logo"
Ad typeFormat classification"native_text", "native_image", "display"

When does indexing happen?

  • When a creative becomes active -- triggered automatically when its status changes to ready/delivering
  • When a creative's image changes -- triggered when an ad image is regenerated
  • Nightly batch -- catches any creatives that were missed or failed during the day
  • Monthly full re-index -- re-analyzes everything to benefit from AI model improvements over time

For native ads, Claude sees both the text (headline, body, call-to-action) and the rendered ad image. For display ads (pre-made banners), Claude only sees the image -- which may contain baked-in text that the AI can read. The landing page URL is also provided for additional context.

The Content Matching Engine

This is the core innovation. When a newsletter is analyzed, its content profile is compared against eligible ad creative profiles to produce match scores -- how well does this ad fit this newsletter?

The problem it solves

Today: Random creative rotation

When a campaign wins an ad slot, the specific creative shown is chosen randomly from eligible options. A Nike running shoes ad and a Nike basketball ad have equal chance of appearing in a basketball newsletter. There's no intelligence matching ad content to newsletter content.

Future: Content-aware weighted rotation

The matching engine scores each creative against each newsletter. The basketball ad gets a high match score for the basketball newsletter. When the ad server picks a creative, it favors higher-scoring matches. No AI call at serve time -- the scores are pre-computed and cached.

How matching works

Not every creative is scored against every newsletter. To keep costs manageable, we pre-filter: only creatives on campaigns targeting this newsletter's placement, only active creatives, and using IAB category overlap as a fast first pass before sending to the AI. This reduces the typical 10,000 creatives down to ~20 per sendout.

Relevance scoring

Rather than rigid formulas (e.g., "topic match = 40%, entity overlap = 25%"), the AI evaluates the overall fit holistically -- considering topic alignment, entity overlap, sentiment compatibility, keyword relevance, and category match all at once. It returns:

0.82
Match score (0 to 1)

Higher = better match

Reasoning

"Strong topical alignment between sports newsletter and athletic gear creative. Brand entities (Nike) appear in both. Positive sentiment matches promotional tone."

Factors

Topic fit: high   Audience alignment: high   Tone match: good   Category overlap: exact

The AI also provides actionable recommendations per creative: "Headline could reference specific NBA content for stronger tie-in," "Consider adding athlete imagery," etc.

Available context signals

What information does the matching engine have to work with?

Important limitation: Subscriber demographics (age, gender, personal interests) are NOT directly available from email providers. The richest signal is the newsletter content itself -- if someone subscribes to a sports newsletter, they're interested in sports. This is why content matching is so valuable.

Future: integration with ad serving

The match scores get cached in Aerospike (the fast in-memory store). When the ad server picks a creative, instead of random shuffle, it does a weighted shuffle -- creatives with higher match scores are more likely to be picked. The ad server already knows the sendout ID, so looking up pre-computed scores is a fast cache read. No AI call happens during ad serving. This is a lightweight code change in one file.

AI-Powered Recommendations

The system doesn't just analyze -- it provides specific, actionable suggestions to improve performance.

🕑

Send Time Optimization

Based on 90 days of hourly performance data, the AI identifies when newsletters perform best. Example: "Your newsletters get 23% higher CTR when sent between 9-10 AM on Tuesdays and Thursdays." Suggests specific scheduling changes.

📸

Creative Fatigue Detection

Tracks each creative's click-through rate over time. When CTR starts declining (e.g., "Summer Sale Banner" down 35% over 2 weeks), it alerts the team and suggests refreshing the imagery, testing alternative headlines, or rotating to backup creatives.

✉

Subject Line Insights

Analyzes which subject line patterns drive engagement. Example: "Newsletters with questions in subject lines show 23% higher open rates" or "Subject lines with specific numbers ('5 Must-See Highlights') outperform vague ones." Contrasts high-performing vs low-performing patterns.

Implementation Timeline

The system is built in 4 phases over ~8 weeks, with a future Phase 5 for ad server integration.

Implementation Phases (Gantt View)
Weeks 1-2

Phase 1: Data Infrastructure

Set up the database tables for storing statistics, analysis results, and creative profiles. Build the API endpoints for retrieving sendout stats. This is the foundation everything else builds on.

Weeks 3-4

Phase 2: AI Integration + Creative Indexing

Connect to the Claude API. Build the newsletter content analysis pipeline. Build the creative indexing pipeline (multimodal -- reads both text and images). Run the initial full index of all 10K active creatives. Hook into creative lifecycle events so new/updated creatives are automatically indexed.

Weeks 5-6

Phase 3: Content Matching + Recommendations

Build the sendout-creative matching engine with pre-filtering. Build relevance scoring. Implement recommendation generators (send time, fatigue, subject line). Create the anomaly detection system with automated alerts.

Weeks 7-8

Phase 4: User-Facing Features

Build publisher dashboard widgets (sendout performance, AI insights feed, creative match scores). Build internal analytics dashboard. Implement scheduled report generation and email delivery for insight digests.

Future

Phase 5: Ad Server Integration

Cache top creative match scores in Aerospike. Modify the ad server's creative selection to use weighted shuffle based on match scores (instead of random). Build A/B test framework to measure the lift. Monitor latency impact.

Technical Safeguards

Performance

AI calls are async

Haiku responds in 1-3 seconds, Sonnet in 3-8 seconds. All AI work runs in background queue jobs -- nothing blocks the user or the ad server.

Statistics are fast

ClickHouse is optimized for aggregation queries. Sendout stats return in under a second, even across millions of events.

Error handling

  • AI down? Ad serving and sendout creation are completely independent of AI. Analysis jobs retry with exponential backoff (3 attempts, then re-queued with delay).
  • Partial failure? If content analysis succeeds but relevance scoring fails, the analysis is still saved. Scoring retries independently.
  • Over budget? Monthly spend is tracked. If it exceeds the threshold, non-critical jobs (insights, reports) pause while core analysis continues.
  • Monitoring: API response times, error rates, and token usage are logged. Alerts fire if sustained error rates exceed 5%.
