1. The bottom line

Verdict

Feasible, and a natural fit for our stack — but the research document is not yet a buildable spec.

The plumbing the product needs (image storage in Google Cloud, server-side redirects, device detection, async workers, analytics pipeline) already exists in Emperor. The proposal correctly identifies animated WebP as the workhorse format. However the document contains at least one factually wrong delivery claim, undercosts Apple Mail's privacy proxy, and skips the three decisions that determine whether this can ship: who runs the transcoder, what we promise to advertisers about measurement, and how we present a single creative across wildly inconsistent email clients.

Engineering can deliver a credible first version with one backend and one frontend engineer, provided the scope is narrowed first.

2. What we'd actually ship — in one sentence

An advertiser uploads a single short video. Our platform produces several lighter versions of it. When a recipient opens their email, our server looks at the email client, picks the best version that client can show, and serves it. If the client can't show motion at all, we serve a still image. Anyone who clicks lands on a hosted page where a real, measurable video plays.

Why this framing matters

It avoids the trap of selling "video in email" as if it were YouTube. In an inbox we can show motion to most recipients, a still to the rest, and recover real video analytics only on the click-through landing page. That is an honest pitch the sales team can repeat without footnotes.

What this means in code paths (technical)
  • Adserver is PHP 8.4 on FrankenPHP worker mode. public/index.php is a worker loop sized for ~20 workers × 40 threads. Long-running transcodes cannot run here — they would starve ad-serving threads. Transcoding lives in a separate daemon.
  • The new ad-display route looks the same shape as the existing HandleDisplay and HandleConversionPixel handlers. One extra Aerospike GET per request to look up which variant matches the requesting UA. The current hot path already does multiple Aerospike GETs; one more is inside the existing budget.
  • The existing GCS upload pipe in the admin API (app/Services/Creative/CreateCreative.php) is multipart-through-API rather than direct-to-storage signed URLs. For a 5–50 MB video source, raising the PHP upload_max_filesize (already 200 MB in php.ini) is enough; we don't need to redesign the upload flow for v1.
  • The runtime container already ships libvpx7 and libwebp7, so adding the ffmpeg binary itself is the change — not a full codec stack rebuild.

4. What the proposal gets right, wrong, and skips

Right

  • Animated WebP as the primary format. Correct call for an inbox dominated by Apple's privacy proxy and Gmail's image proxy.
  • Server-side detection and 302 redirect (Strategy A). Direct fit for the redirect plumbing we already use for pixel tracking and click tracking.
  • The ffmpeg command reference. The encoding settings (baseline H.264 for iOS webview, +faststart for progressive playback, animated WebP at 15fps) are accurate and would survive code review.
  • Apple MPP indistinguishability. Acknowledging that iPhone, iPad and Mac look the same when fronted by MPP is the right framing.

Wrong

Blocker The <picture> fallback as described does not exist

The proposal (line 30) suggests using <picture> with <source type="video/webm"> and image fallbacks, claiming the client picks what it supports with no JS. This is not how the spec works. The <picture> element selects between image sources by MIME type — video/* types are not valid in that context, and no browser, let alone an email client, will render a <video> from inside <picture>.

The author appears to have confused <picture><source> (image negotiation) with <video><source> (codec negotiation inside a video tag). Even using the correct <video> element, Gmail web, Outlook.com and Yahoo aggressively sanitize <video> as of late 2025 — so the only client that meaningfully honors this path is Apple Mail.

Fix: drop Strategy B from the proposal, or rewrite it as "Apple Mail desktop only, in a <video> tag with an <img> fallback inside it." Strategy A (server-side redirect) covers every other environment.

Major "Outlook desktop = static poster" is stale

That statement was true for the legacy Word-renderer Outlook (Outlook 2016–2021, classic 365). The new Outlook for Windows, rolled out broadly through 2024–2025, is WebView2-based and renders much closer to Outlook.com than to the classic client. Treating "Outlook desktop" as a single row in 2026 understates the audience that can actually see animation.

Fix: split the matrix into Outlook Classic (Word) → static poster, and new Outlook / Outlook 365 web stack → animated WebP. UA detection for new Outlook is imperfect — acknowledge that gap explicitly.

Minor Header signals are listed more confidently than they hold up

The Accept header sends video/* only on <video> fetches, not on <img>-style requests — and the creative is fetched as an image in nearly every email context. Sec-Fetch-Dest is sent by browsers, not by native mail clients (Apple Mail desktop, Outlook desktop). Both signals are useful in narrow cases but should not be presented as primary classifiers.

Skipped (what's missing)

Blocker No execution model for transcoding

The proposal recommends ffmpeg six different ways but never says where it runs, who installs it, or how a transcode job is dispatched, monitored, or retried. ffmpeg is not in our adserver container today; adding it brings a new binary, a larger image, a wider attack surface for user-uploaded video, and H.264/AAC patent licensing exposure for commercial distribution.

Without a transcoder execution model in the doc, a build team will inherit a fait accompli on day one. This must be decided up front — see Section 6.

Blocker No model of what Apple MPP costs us

Apple Mail Privacy Protection prefetches every image at message open from Apple's egress, once per recipient, whether the human ever looks at the email. For 1M MPP recipients and a 463 KB animated WebP, that is roughly 450 GB of egress per send, attributed to "opens" not "views." At Google Cloud Storage multi-region egress rates this is real money, and it happens whether the campaign performs or not.

The product decision is: serve MPP recipients the animated WebP (rich, expensive, looks great in 80% of cases), or the static poster (cheap, no motion). Picking one shapes the entire commercial story.

Major No measurement story for sales

Tracking in the platform is open + click only, no JavaScript runs inside the email creative itself. We cannot tell whether the video played, looped, or rendered as nothing more than a poster frame. MPP prefetch further pollutes "open" numbers. The proposal acknowledges this constraint in passing but never tells sales what numbers they can actually quote to advertisers.

Fix: enumerate the metrics that do exist (variant-served counts by format, click-through, post-click landing-page dwell time and instrumented playback). State plainly that "video view %" as advertisers expect it does not exist in-inbox; it exists only on the click-through landing page.

Major No asset lifecycle, no transcode state machine

Six variants per creative, multiplied by creatives × resends × re-encodes, is a storage bucket that grows without rules. There is no retention policy, no cold-storage transition, no content-addressed naming, and no story for what happens between upload and "creative is ready to serve." If a trafficker uploads at T+0 and tries to launch a campaign at T+30 seconds, does the campaign serve the poster? Fail validation? Wait?

This must be a defined state machine: uploaded → transcoding → partially_ready (poster only) → ready (all variants) → failed, with explicit rules for which states allow activation.

5. The four building blocks

What we'd actually build, broken into four pieces in plain language. Each accordion has the engineering detail.

A. Upload page — admin and client apps

A new section on the creative-form lets an advertiser drag in one video file, see a small preview, watch a progress bar while transcoding runs, and inspect the generated variants before saving. Looks and feels like our existing image upload, with a transcode-status badge.

Upload — technical detail
  • New component RHFVideoUpload mirrors RHFImageUpload (React Hook Form + react-dropzone) but accepts video/mp4, video/quicktime, max ~50 MB.
  • Existing CreativeForm.tsx gains a creative-type toggle (image vs video). Video mode routes to the new component.
  • Three new sibling components: TranscodeStatusBadge (poll status), VideoVariantInspector (list generated assets), VideoFallbackConfig (MVP: read-only display of routing rules).
  • API endpoint extends CreateCreative.php with a video branch — uploads source to GCS at passendo-creative-files/videos/raw/{groupId}.mp4, emits a Kafka message video.transcode.requested, returns immediately with status transcoding.
  • The existing image editor (Polotno) does not participate in the v1 video flow — see Section 7.
B. Transcoder — a new background worker

A new background worker picks up uploaded videos from a queue, produces the lighter versions our platform needs (animated WebP, MP4, WebM, GIF, JPG poster), stores them in Google Cloud Storage, and marks the creative ready. It runs separately from the live ad-serving traffic so it cannot slow anything down.

Transcoder — technical detail
  • A seventh daemon (daemon7.sh → process:video-transcode Symfony console command) consumes the new Kafka topic video.transcode.requested.
  • Lives in a separate container image (Dockerfile.daemon-video) so ffmpeg's ~200 MB footprint is isolated from the FrankenPHP hot path.
  • Runs the exact ffmpeg invocations from the proposal (lines 75–95) — they're correct.
  • Writes one row per variant into a new creative_variants table, emits a creative.status = ready event.
  • Patent question on H.264/AAC encoding must be resolved by legal before the worker ships externally. Mitigation if needed: ship animated WebP + WebM only in v1, defer MP4 to v2.
C. Smart delivery — adserver picks the right version

When the recipient's email client opens the message, the adserver looks at the request, decides which version of the video this client can handle, and sends it. If it can't tell, it sends the still image. Built on the exact same redirect pattern our platform already uses for clicks and tracking pixels.

Smart delivery — technical detail
  • New route on the adserver: /vdisplay/{creativeGroupId}/{layoutId}/{orientation}, modeled on HandleDisplay.php.
  • Inspects the request via the existing CreateDeviceUsingParser — already classifies Apple MPP, Apple non-MPP, Gmail proxy.
  • Looks up the variant set in Aerospike (cv:{groupId}:{layout}:{orient}) — one extra GET per request, within the current hot-path budget.
  • Returns a 302 redirect to a signed GCS URL for the chosen variant. Time-to-first-byte is dominated by Aerospike + UA parse, both already sub-millisecond.
  • If classification confidence is low, default to the static poster JPG — fail-safe behaviour.
D. Reporting — what shows up in the dashboards

One new column on the standard creative-reporting view: which version was served. Sales can break a campaign down by "how many recipients got the animated version vs the still." No new column for "video views" — we cannot honestly produce that number from an email open alone.

Reporting — technical detail
  • Two new columns on events in ClickHouse: creativeGroupId String and creativeVariantFormat LowCardinality(String) — follows the multi-CTA schema-extension precedent already in the repo.
  • Reporting tables in the FE get one new optional column.
  • The hosted click-through landing page is the only place we get real video-completion metrics. v1 ships with a basic JS-instrumented landing page; v2 wires those numbers back into the creative dashboard.

6. Build vs buy — where does the transcoding happen?

This is the single most important decision in the project. Three options:

  1. Self-hosted ffmpeg in a new background worker (the architecture lane's recommendation).
  2. Cloud transcoder (Mux, Transloadit, Cloudflare Stream, AWS MediaConvert).
  3. Hybrid: animated WebP via Imagick (already installed), everything else outsourced.
Recommendation: self-hosted ffmpeg in a new worker container

Same Kafka/console pattern as our six existing daemons. Full control over animated WebP — the format the proposal calls the workhorse, and the one most cloud transcoders don't reliably support. Image bloat is contained because the worker runs in a separate container from the ad-serving hot path.

7. Where Polotno fits — and where it doesn't

You asked the right question. The proposal as written is upload-only: an advertiser hands us a finished video file and we transcode it. Polotno — the in-browser canvas editor we use to compose image creatives — is not in the flow at all. That creates a real UX inconsistency: image creatives are designed in-browser; video creatives have to be designed outside Passendo and uploaded as a finished asset.

The honest answer

Polotno cannot produce video. It exports stills (toDataURL) and animated GIFs assembled from frame stills (toGIFDataURL) — no MP4, no WebM, no animated WebP. So a Polotno-driven "compose your own video creative" flow is not a thin extension of what we already have; it would be a significant new build.

Three sensible positions, in order of scope:

  1. Stage 0 (recommended for v1): Polotno does not participate. Advertisers upload a finished MP4. Document this as a deliberate scope choice in the spec.
  2. Stage 1 (v1.5 or v2): Polotno is used to design the poster frame and end-card only. Polotno exports a still, ffmpeg overlays the still as the first/last frame of the video, advertiser sees a unified design experience. Modest ffmpeg work; reuses Polotno as-is.
  3. Stage 2 (long-horizon): Polotno is used to design an overlay (logo, text, CTA box) that the transcoder bakes into the video frame-by-frame using the Polotno JSON as a recipe. Real engineering; needs careful design.
Polotno integration detail (technical)

What Polotno does today in our app

  • Located at frontend/apps/client/src/components/polotno/Polotno.tsx (719 lines, package version 3.0.0-9).
  • Canvas composition with text, elements, layers, image upload, template save/load. Templates fetched via useFullImageTemplatesList.
  • Export surface: store.toDataURL() for stills, store.toGIFDataURL() for animated GIFs. store.toJSON() serializes the template.
  • No video output exists in the Polotno API surface we use, and not in the upstream library either.

What would be needed for Stage 1 (poster/end-card composer)

  • Add a "Design poster" step to the video creative form that opens the existing Polotno editor at the source video's dimensions, prefilled with a still extracted from the uploaded MP4 (one ffmpeg -frames:v 1 call by the transcoder).
  • Polotno exports the designed poster as a PNG/JPG.
  • Transcoder takes the original MP4 + the designed poster and produces variants where the poster is the first frame (and optionally the last, as an end-card hold).
  • No changes to Polotno itself; modest FE + transcoder glue.

What would be needed for Stage 2 (overlay compositor)

  • Polotno produces a layered JSON template (text + logo + CTA box) with no underlying image.
  • Transcoder converts that JSON into an ffmpeg overlay filter chain or renders each Polotno layer as a transparent PNG and overlays it via ffmpeg.
  • Significant work: font fidelity across our font set, layout against varying source aspect ratios, performance per encode. Defer until v1 ships and there is a real demand signal.

8. What we can — and can't — measure

This is the part the proposal under-served, and it directly affects what sales can promise.

What we get
  • Variant served: how many recipients received which version
  • Opens (polluted by Apple MPP prefetch)
  • Clicks
  • Landing page playback metrics (real <video> events with JS)
  • Click-through dwell time on the hosted page
What we do not get inside the email
  • Did it play?
  • How long did they watch?
  • Did it loop?
  • Was sound enabled? (always muted by spec)
  • Did they actually see motion, or just the poster frame?

The sales pitch should be: rich animated creatives in the inbox with guaranteed static fallback, and full video analytics on the click-through landing page. Not: video in email with measurable view rates.

9. Risks worth naming

  • New ffmpeg dependency. Container bloat, build time, security surface for arbitrary uploads. Manageable; contain it in a separate worker image.
  • H.264/AAC patent licensing for self-encoded MP4 in a commercial product. Real, bounded, needs legal sign-off before v1 ships.
  • Apple MPP egress economics. A 1M-recipient send with a 463 KB animated WebP variant is ~450 GB of GCS egress, charged whether anyone watches. Decide upfront whether MPP recipients get the animated WebP (rich, expensive) or the poster JPG (cheap, no motion).
  • Detection misfires. If UA classification is wrong and we serve MP4 to Outlook Classic, the recipient sees a broken image. Fail-safe default to the JPG poster when confidence is low.
Full risk list (technical)
  • Transcode latency vs. campaign go-live — define the state machine.
  • Re-encode of an updated source while live campaigns are serving the old version — versioning rules.
  • Aerospike extra GET per ad-request — needs a perf test, the locustfile.py setup already exists.
  • Content policy: autoplay-in-inbox, photosensitive epilepsy guard, max brightness/flash, alt text for screen readers on the <img> fallback.
  • Storage lifecycle: GCS Standard → Nearline → Coldline transitions for old variants. No rule defined today.
  • CDN cache key includes UA classification → cardinality implications if classes proliferate.
  • Rejection / QA flow for 1080p HEVC sources at 200 MB — hard reject? auto-downscale? manual review?
  • Apple MPP indistinguishability — we cannot tell iPhone vs Mac vs iPad through MPP, so per-device sizing is not possible for those recipients.
  • Frontend upload UX: chunked / resumable uploads will eventually be needed for >50 MB sources over weak networks; defer to v2.

10. Smallest credible MVP

Recommended MVP (one sentence)

One MP4 upload (≤50 MB, ≤30s) → five variants produced in a new daemon → one new adserver route doing UA-based 302 redirect → one new column in reporting. No editable fallback config. No Polotno integration. No <picture> tag. Defer everything else to v2.

11. Open decisions for stakeholders

These are not engineering questions. They are product and commercial calls that engineering should not make alone.

  1. Apple MPP variant policy. Animated WebP (rich, expensive) or static poster (cheap, no motion) for the largest single recipient segment in many campaigns? Shapes the cost model.
  2. Legal sign-off on H.264/AAC encoding for commercial distribution. If sign-off is slow, v1 ships animated WebP + WebM only.
  3. Is the click-through hosted landing page in scope for v1? Without it, sales has no real video-view numbers to quote. With it, v1 scope grows.
  4. Polotno integration roadmap. Confirm v1 is upload-only (no editor), v1.5 considers Polotno for poster/end-card design, v2 considers overlay composition.
  5. Duration cap. Pick a single number, not a range. Recommend 30s hard cap at upload validation.
  6. Maximum file-size cap at upload. Recommend 50 MB; advertisers wanting more get directed to the hosted landing-page product.
  7. Brand-safety toggle for advertisers who do not want motion (force static even where animation is supported).

12. Appendix — full reports

Two specialist reviews informed this document. Both are reproduced below in full for the engineering audience.

Full critic report (verdict: revise; one Blocker, three Major findings)

The critic lane produced a 1,500-word adversarial review of the proposal. Highlights:

  • Blocker 1: the <picture><source type="video/webm"> claim is factually wrong; a build team would attempt to implement it.
  • Blocker 2: ffmpeg is recommended but the dependency is unowned — no execution model, no patent posture, no operational plan.
  • Blocker 3: Apple MPP unit economics are ignored; one defensible cost model required before stakeholders sign off.
  • Major findings: Outlook desktop matrix is stale (new Outlook = WebView2); <picture> "client picks" claim overstates webmail capability; analytics story is missing; storage variant fan-out unowned; transcode latency vs. campaign go-live undefined; FE upload UX scope omitted.
  • Minor findings: Accept header reliability overstated, Sec-Fetch-Dest not sent by native mail clients, file-size table caveats buried, duration cap is a range not a number.
  • Recommended cut: strip to animated WebP + static JPG poster + hosted click-through landing page. Defer MP4, WebM, GIF to v2. Eliminates the patent question and bounds the MPP cost in v1.

The critic explicitly verified the following from the codebase before writing: composer.json ships Imagick but not ffmpeg; src/Click/Redirect/ and src/Device/CreateForGmailProxy.php confirm redirect + proxy-classification plumbing is real; RHFImageUpload exists but is image-only. Verdict was REJECT (REVISE) — the bones of a workable plan are present, but the proposal must own the architectural and commercial questions it currently dodges before a build team can be tasked.

Full architecture / feasibility report (verdict: feasible; build-don't-buy)

The architecture lane produced a 2,000-word feasibility assessment grounded in code paths. Highlights:

  • Most critical finding: libvpx7 and libwebp7 are already installed in the adserver runtime (Dockerfile), so adding ffmpeg is the binary install — not a codec stack rebuild.
  • Hot-path verdict: the new variant lookup adds one Aerospike GET per ad-request. The current path already does multiple GETs; the extra one fits the existing budget. Needs a perf test (the project already has a locustfile.py).
  • Transcoder placement: cannot run inside the FrankenPHP worker (would block ad-serving threads). Must run as a seventh daemon (daemon7.sh) consuming a new Kafka topic — identical to the six existing daemons.
  • Build vs buy: self-hosted ffmpeg in a separate worker container. Cloud transcoders either don't output animated WebP (the workhorse format) or use the wrong delivery model (HLS instead of static asset).
  • Storage model: add a creative_variants table; variants are siblings keyed by creative_group_id. No change to the existing creatives table.
  • FE work: RHFVideoUpload sibling to RHFImageUpload; new components TranscodeStatusBadge, VideoVariantInspector, VideoFallbackConfig. Creative-form gains a type toggle. Existing creatives-image-editor is canvas-image-specific and is not reusable for video.
  • Effort shape: one BE + one FE engineer. Critical path is API upload contract → transcoder daemon → adserver route. FE upload, reporting, and rollout work parallelize once the API contract is frozen.
  • Honest gaps: the architect could not verify the contents of the passendo/native-image-builder vendor library (empty in this checkout) or read the live ClickHouse events table DDL. Confirm both with the team before locking the daemon design.
