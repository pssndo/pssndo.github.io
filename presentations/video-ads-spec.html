Video Ads in Email Newsletters

Epic: Video-Like Ad Experiences Within Email

Overview

Email clients do not support <video> tags, autoplay, or embedded video players. This is a hard limitation of the email rendering environment — Gmail, Outlook, Apple Mail, and Yahoo all strip or ignore video elements. There is no way to serve a real video player inside an email.

However, there are proven techniques to deliver video-like ad experiences that feel close to watching a video, using formats that work across all major email clients.

This document describes what's possible, what needs to change in Emperor, and the recommended approach.

Apple Mail is the only client with partial <video> support, but it requires user interaction to play and covers ~15-20% of email opens. Building for that alone is not viable as a primary strategy.

Other formats considered but not recommended:

  • APNG (Animated PNG): Better quality than GIF (full 24-bit color, no banding) but inconsistent email client support. Would require GIF fallback for Outlook, adding complexity without significant benefit. If we pursue this later, the pipeline is nearly identical to GIF — just a different FFmpeg output format.

What We Can Do: Two Approaches

RECOMMENDED Approach 1: Animated GIF Ads

What it is: A short animated GIF (3-8 seconds, 5-15 frames) that auto-plays in the email. Looks and feels like a silent looping video clip.

Why it works:

  • GIFs are supported by every email client (even Outlook falls back to showing the first frame)
  • Auto-plays on load — no user interaction needed
  • Already image-based — fits directly into Emperor's existing <img src="/imp/..."> flow

What the advertiser provides:

  • A short video file (MP4/MOV, 3-8 seconds) or a pre-made animated GIF
  • Alternatively: a sequence of images (frames) that we assemble into a GIF

What changes in Emperor:

ComponentChange
Creative modelNo new ImageSourceType needed — GIFs can already be stored as images. If we need to store the source video file for re-processing, add VIDEO type. Consider adding isVideo boolean flag for reporting/filtering purposes.

Limitations:

  • No sound (emails never have sound)
  • Outlook desktop shows only the first frame (static fallback — this is fine, ~10% of opens)
  • File size constraint: large/long GIFs get heavy fast. 3-5 seconds at 600px wide is the sweet spot.
  • Color banding on gradients (GIF is 256 colors max)

Approach 2: Interactive Thumbnail + Landing Page Video (Click-to-Play)

What it is: A static or animated thumbnail image in the email with a prominent play button overlay. Clicking opens the landing page where the full video plays.

Why it works:

  • Works in 100% of email clients (it's just an image + link)
  • No file size concerns for the email itself
  • Full video experience on the landing page (no email restrictions)
  • Higher intent signal — user chose to watch

What the advertiser provides:

  • Video file (hosted on their landing page or YouTube/Vimeo)
  • Thumbnail image (or we auto-generate from video first frame)

What changes in Emperor:

ComponentChange
Creative modelNo new ImageSourceType needed — this is essentially a standard image creative with a landing page URL. Consider adding isVideo boolean flag for reporting/filtering if we want to track video ads separately.
Play button overlayNativeImageBuilder composites a play button icon (triangle in circle) over the thumbnail image. This is a standard ImageMagick composite operation on a single frame.
Landing page URLThe click URL redirects to the video landing page (already supported — it's just landingPageUrl).
Creative upload (API)Accept video URL (YouTube/Vimeo/direct) + optional thumbnail. If no thumbnail provided, generate one from the video's first frame via FFmpeg.

This approach requires the least code changes — it's essentially a regular display/native ad with a play button overlay on the image. No new types needed.

Recommended Implementation Plan

Phase 1: Click-to-Play Thumbnail (Lowest Effort, Immediate Value)

This is the fastest path to "video ads in email":

  1. Add play button overlay capability to NativeImageBuilder (composite a semi-transparent play icon over the creative image).
  2. Optionally add isVideo boolean flag to creative model for reporting/filtering purposes.
  3. Advertiser sets landingPageUrl to their video landing page.
  4. Frontend: add "Video Ad" option when creating a creative. Shows thumbnail upload + video landing page URL fields.
  5. Deploy — video thumbnail ads work immediately in all email clients.

Effort: Small. It's an ImageMagick composite operation + optional flag. No new ImageSourceType needed — this is fundamentally a standard image creative with a landing page.

Phase 2: Animated GIF Ads (Full Video-Like Experience)

  1. Optionally add VIDEO to ImageSourceType enum if we need to store the source video file for re-processing. Otherwise, just store the converted GIF as a regular image.
  2. Install FFmpeg in API container (or use a dedicated media processing container).
  3. Implement ConvertVideoToAnimatedGif queue job:
    • Accept MP4/MOV/WebM upload (max 15s, max 20MB)
    • FFmpeg pipeline: extract frames → optimize palette → generate GIF → validate size
    • If GIF exceeds 2MB: reduce resolution/fps and retry (up to 3 attempts)
    • Store result in GCS
  4. Update display route to serve GIF with Content-Type: image/gif. Note: Verify if content type is already set dynamically based on image format — may require no changes.
  5. Add creative validation: reject videos longer than 15 seconds or files larger than 20MB.
  6. Frontend: video upload field with preview, duration indicator, file size warning.

Effort: Medium. FFmpeg integration + conversion pipeline + file size optimization loop.

DEFERRED Phase 3: Hybrid Native GIF Ads (Text Overlay on GIF)

Deferred until layout editor refactor (Polotno or in-house).

  1. Extend NativeImageBuilder to handle animated GIF input:
    • Decompose GIF into frames
    • Composite headline/CTA/adIcon on each frame
    • Reassemble into output GIF
  2. This lets advertisers upload a video and Emperor renders it as a native ad with consistent branding — same headline/CTA treatment as static native ads.

Effort: High. Frame-by-frame ImageMagick processing is complex and CPU-intensive. This depends on the new layout builder — revisit after that refactor is complete. For now, require advertisers to bake text into their videos.

FFmpeg Conversion Pipeline (Phase 2)

# Step 1: Generate optimized palette
ffmpeg -i input.mp4 -vf "fps=8,scale=600:-1:flags=lanczos,palettegen=max_colors=192" palette.png

# Step 2: Create GIF using palette
ffmpeg -i input.mp4 -i palette.png -lavfi "fps=8,scale=600:-1:flags=lanczos[x];[x][1:v]paletteuse=dither=bayer:bayer_scale=3" -loop 0 output.gif

# Step 3: Validate size
# If output.gif > 2MB, retry with fps=5 and/or scale=400:-1

PHP implementation wraps this in a process call with timeout and output validation.

Display Route Changes (Adserver)

/display/{creativeId}/{layoutId}/{orientation} currently always returns Content-Type: image/png.

Question to verify: Does the display route already set Content-Type dynamically based on the uploaded image format? If so, no changes may be needed for serving GIFs.

If changes are needed:

If image is GIF:
    → Return GIF blob with Content-Type: image/gif, Cache-Control: public, max-age=1800
If video thumbnail (play button overlay requested):
    → Generate PNG with play button overlay via NativeImageBuilder, return as image/png
Otherwise:
    → Current behavior (PNG)

File Size Budget

Email providers have total message size limits. Ad images compete with newsletter content for this budget:

Target: 2 MB max for animated GIF ads. This is achievable for 3-5 second clips at 600px width with palette optimization.

For comparison, current static PNG ads are typically 50-200 KB.

Acceptance Criteria

Phase 1 (Click-to-Play Thumbnail)

  • Play button overlay rendered on thumbnail image by NativeImageBuilder.
  • Click redirects to video landing page URL.
  • Works in all email clients (it's a standard image + link).
  • Frontend shows "Video Ad" creative option with thumbnail + landing URL fields.
  • Optional: isVideo flag available for reporting/filtering purposes.

Phase 2 (Animated GIF)

  • Video upload accepts MP4/MOV/WebM (max 15 seconds, max 20MB).
  • FFmpeg converts video to optimized animated GIF.
  • Generated GIF is under 2MB (auto-reduces quality if needed).
  • /display/ route serves GIF with correct content type (verify if already supported).
  • GIF auto-plays in Gmail, Apple Mail, Yahoo Mail, iOS Mail.
  • Outlook shows first frame as static fallback.
  • Creative validation rejects videos over 15 seconds.
  • Optional: VIDEO image source type if we need to preserve original video file.

Phase 3 (Hybrid Native GIF) — Deferred

  • Blocked on: New layout builder (Polotno or in-house).
  • NativeImageBuilder can overlay headline/CTA on animated GIF frames.
  • Output GIF has consistent branding across all frames.
  • File size remains under max limit after text overlay.
